# 問題作成計画

## 目的

公開アプリ向けに、各カテゴリ100問へ拡張しても品質を維持できるよう、作問工程を先に標準化する。

このドキュメントでは以下を定める。

- 作問の進め方
- レビュー基準
- `part1` の100問配分表
- 台帳運用ルール

## 前提

- 現状データは [assets/quiz_data.json](/Users/kurok/StudioProjects/02.sikaku/drone1/assets/quiz_data.json) にある
- 現状の `part1` は26問
- 公開アプリのため、法令・制度に関する誤りは許容しない
- 問題は最終反映前に必ず台帳で管理する

## 基本方針

1. 問題は直接JSONへ追加しない
2. 20問単位で作成し、20問ごとにレビューする
3. レビュー通過分のみを確定問題として扱う
4. 100問完成後にカテゴリ単位で通し監査する
5. 通し監査後にのみJSONへ反映する

## 品質基準

### 1. 正確性

- 法令、制度、数値、期限、条件のある設問は根拠ソース必須
- 法令系は国土交通省の公式情報を基準にする
- 根拠確認日を必ず残す

### 2. 一意性

- 1問1論点に限定する
- 読み手が複数解釈できる文を避ける
- 「原則」「例外」「条件付き」を混同しない

### 3. 学習価値

- 暗記だけでなく、運用判断に効く問題を含める
- 似た言い換え問題で水増ししない
- 誤答時に学びが残る解説にする

### 4. バランス

- 正誤比はカテゴリ内で `正50 / 誤50` を目標にする
- 難易度は `初級40 / 中級40 / 上級20`
- 中分類ごとに偏りを管理する

### 5. 保守性

- すべての問題にIDを振る
- 台帳上でレビュー状態を管理する
- 法改正や制度変更に備え、根拠確認日を残す

## 作問フロー

1. 中分類ごとに必要数を決める
2. 既存問題を中分類へ再配置する
3. 不足分のみ新規作問する
4. 20問ごとに自己監査する
5. 20問ごとにカテゴリ内重複チェックを行う
6. 100問到達後に通し監査する
7. JSON反映前に表記統一を行う

## レビュー観点

- 根拠ソースが明記されているか
- 設問が1論点に収まっているか
- 正誤が明確に判定できるか
- 解説が「なぜそうなるか」を説明しているか
- 既存問題と論点重複していないか
- 難易度設定が適切か
- 初学者に誤解を与える悪文になっていないか

## part1 既存26問の扱い方針

### そのまま活用しやすい論点

- 無人航空機の定義
- 登録の有効期間
- 登録記号の表示
- リモートID
- DID
- 150m以上
- 緊急用務空域
- 夜間飛行
- 目視外飛行
- カテゴリーIII
- 一等/二等
- 技能証明の有効期間
- 飛行日誌
- 事故報告

### 追加が必要な論点

- 催し場所上空
- 人又は物件との30m未満飛行
- 危険物輸送
- 物件投下
- 補助者配置と立入管理措置の切り分け
- 機体認証と技能証明の組み合わせ
- 許可承認が不要になる条件
- DIPS申請実務
- リモートIDの例外条件
- 事故と重大インシデントの切り分け

## 台帳運用ルール

- 台帳の `status` は `draft` `reviewed` `approved` `rejected` を使う
- `source_type` は `law` `mlit_page` `manual` `guideline` を使う
- `duplicate_group` は類題管理用に使う
- JSONへ反映できるのは `approved` のみ
- `review_notes` が空のまま `rejected` にしない

4. 20問ごとにレビューする
5. 100問到達後にカテゴリ監査する
6. JSON反映方針を決めてアプリへ投入する
